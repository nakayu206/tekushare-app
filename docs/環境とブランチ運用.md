## 1. Flutter のビルドモード（Debug / Profile / Release）

Flutter には目的別に 3つのビルドモードがあります。

### Debug（開発中）

- ✅ ホットリロードが使える
- ✅ 詳細なスタックトレース
- ❌ 動作が遅い／アプリサイズ大

→ 開発者が「コードを書いて確認する」ための環境。

### Profile（性能計測）

- ✅ Release に近い速度
- ✅ DevTools で FPS・メモリ・CPU 計測可能
- ❌ エミュレータ不可（実機必須）

→ 「カクつかないか？」「メモリ食いすぎてないか？」をチェック。

### Release（本番配布）

- ✅ 最大限に最適化／アプリサイズ最小
- ✅ デバッグ情報なし（セキュリティ的にも安全）
- ❌ ホットリロード・デバッグ不可

→ ストア公開や実ユーザーが使うバージョン。

### まとめ

| モード  | 速度 | デバッグ | ホットリロード | 用途           |
| ------- | ---- | -------- | -------------- | -------------- |
| Debug   | 遅い | ◎        | ◎              | 開発中         |
| Profile | 速い | △        | ✗              | 性能測定       |
| Release | 最速 | ✗        | ✗              | ユーザーに配布 |

> **「開発しやすさ」と「速さ」は両立できないので、目的別にモードを切り替えている**

## 2. 環境（Flavor: dev / stg / prod）

ビルドモードとは別に、**接続先や設定を切り替える仕組み**として「Flavor」があります。

### dev（Development / 開発環境）

- 接続先：開発用サーバー（テストデータ）
- DB を吹き飛ばしてもOK
- 開発者が自由に壊していい環境

### stg（Staging / ステージング環境）

- 接続先：本番とほぼ同じ構成
- データも本番に近い
- QA・テスター・社内レビューで使う

### prod（Production / 本番環境）

- 接続先：本番サーバー（実データ）
- 絶対に壊してはいけない
- App Store / Google Play で配信

### なぜ分けるか

| 困ること                 | 例                                        |
| ------------------------ | ----------------------------------------- |
| 本番データが壊れる       | バグでユーザーの注文を消してしまう        |
| テストできない           | 決済機能を本番でテスト = 実際にお金が動く |
| リリース前確認ができない | 本番に出してから不具合発覚                |

```
開発者 → [dev] でガンガン作る
   ↓
QA   → [stg] で本番同等の環境で検証
   ↓
ユーザー → [prod] で安心して使える
```

### このプロジェクトでの実装

| ファイル             | Flavor | 用途   |
| -------------------- | ------ | ------ |
| `lib/main_dev.dart`  | dev    | 開発用 |
| `lib/main_stg.dart`  | stg    | 検証用 |
| `lib/main_prod.dart` | prod   | 本番用 |

iOS 側にも対応する xcconfig：

- `ios/Flutter/Dev.xcconfig`
- `ios/Flutter/Stg.xcconfig`
- `ios/Flutter/Prod.xcconfig`

## 3. テストはどの環境でやるべきか

「テスト = stg」とは限りません。テストの種類によって場所を変えます。

| テストの種類             | やる場所 | 誰が         | 何を確認？                 |
| ------------------------ | -------- | ------------ | -------------------------- |
| ユニットテスト           | ローカル | 開発者       | 関数1つが正しく動くか      |
| ウィジェットテスト       | ローカル | 開発者       | UI部品が正しく表示されるか |
| 結合テスト（開発中）     | **dev**  | 開発者       | 機能を作りながら動作確認   |
| QA / 受け入れテスト      | **stg**  | QA・テスター | 本番同等の環境で総合確認   |
| 本番動作確認（スモーク） | **prod** | 開発者・運用 | リリース後に最低限動くか   |

### フロー

```
[コード書く] → dev で動作確認 → stg で最終チェック → prod へリリース
   ↑              ↑                 ↑
ローカルで        開発者が           QAが
unit/widget       手動で確認         本番同等で検証
テスト
```

### コマンド例

```bash
# 開発中の動作確認
flutter run --flavor dev -t lib/main_dev.dart

# リリース前のテスト
flutter run --flavor stg -t lib/main_stg.dart

# 本番ビルド
flutter run --flavor prod -t lib/main_prod.dart
```

## 4. 環境ごとに Git ブランチを分けるべきか

### 結論

> **環境ごとに長命ブランチを作る運用（dev/stg/prod ブランチ）は非推奨。**
> **「単一の main ブランチ + Flavor + CI/CD」が業界標準。**

### 根拠

#### 1. GitHub 公式（GitHub Flow）

[GitHub Docs - GitHub flow](https://docs.github.com/en/get-started/using-github/github-flow)

- `main` ブランチは常にデプロイ可能
- 機能ごとに短命な feature ブランチ
- 環境別ブランチの記述は無い

#### 2. Git Flow 提唱者本人の追記（2020年）

[Vincent Driessen - A successful Git branching model](https://nvie.com/posts/a-successful-git-branching-model/)

> "If your team is doing continuous delivery of software, I would suggest to adopt a much simpler workflow (like GitHub flow) instead of trying to shoehorn git-flow into your team."

→ Git Flow を作った本人が「継続的デリバリーするなら GitHub Flow を使え」と明言。

#### 3. ThoughtWorks Technology Radar

[Gitflow（HOLD判定）](https://www.thoughtworks.com/radar/techniques/gitflow)

> "We recommend against this pattern, favoring trunk-based development instead."

#### 4. Google DORA（State of DevOps Report）

[Trunk-based development](https://cloud.google.com/architecture/devops/devops-tech-trunk-based-development)

高パフォーマンスチームの特徴：

- 3本以下のアクティブブランチ
- ブランチの寿命が 1日未満
- main から直接デプロイ

#### 5. Martin Fowler

[Patterns for Managing Source Code Branches](https://martinfowler.com/articles/branching-patterns.html)

> "Environment Branch is a bad idea, it's better to make any environment-specific changes through configuration."

#### 6. The Twelve-Factor App

[III. Config](https://12factor.net/config)

> "Strict separation of config from code."

→ 環境差は**コード（=ブランチ）ではなく設定で分離**するのが原則。

### 環境別ブランチのデメリット

- 😱 マージ漏れ（stgで直したのにprodに反映されてない）
- 😱 ブランチ間の差分でバグが起きる
- 😱 hotfix を3つのブランチ全部に当てる必要がある
- 😱 「どのブランチに何が入ってる？」が不明瞭

## 5. main 1本から stg / prod をどう出し分けるか

ブランチでは分けず、**「タグ」または「リリースブランチ」+「CI/CD」**で出し分けます。

### 🌟 パターン1：タグでリリース（最推奨・小〜中規模）

```
main ──●──●──●──●──●──●──●──
       │     │        │
       ▼     ▼        ▼
     [自動]  v1.0.0  v1.1.0   ← タグ = prod リリース
      stg   (prod)   (prod)
```

| トリガー             | デプロイ先              |
| -------------------- | ----------------------- |
| `main` への push     | **stg に自動デプロイ**  |
| `v*.*.*` タグを push | **prod に自動デプロイ** |

#### 運用フロー

```bash
# 1. 機能開発
git checkout -b feature/login
git push origin feature/login
# → PR → レビュー → main マージ → 自動で stg にデプロイ

# 2. stg で QA テスト OK！

# 3. 本番リリース
git checkout main
git pull
git tag v1.0.0
git push origin v1.0.0
# → 自動で prod にデプロイ
```

#### メリット

- ブランチ1本で済む
- リリース履歴がタグで一目瞭然
- ロールバックも `git checkout v0.9.0` で簡単

### 🔧 パターン2：release ブランチ（中〜大規模・厳格運用）

[Microsoft Release Flow](https://learn.microsoft.com/en-us/devops/develop/how-microsoft-develops-devops) の方式。

```
main    ──●──●──●──●──●──●──●──
              │           │
              └─ release/1.0  ──●──●──  ← prod
                                       └─ hotfix
              [stg は main から自動]
```

| ブランチ      | デプロイ先     | 寿命               |
| ------------- | -------------- | ------------------ |
| `main`        | stg（自動）    | 永続               |
| `release/x.y` | prod（タグで） | リリースごと・短命 |

### 比較表

| 方式             | ブランチ数        | stg→prod 切替        | 難易度 | 推奨度     |
| ---------------- | ----------------- | -------------------- | ------ | ---------- |
| タグ運用         | main のみ         | タグを打つ           | ⭐     | ⭐⭐⭐⭐⭐ |
| release ブランチ | main + release/\* | release ブランチ作成 | ⭐⭐   | ⭐⭐⭐⭐   |
| 環境別ブランチ   | dev/stg/main      | ブランチ間マージ     | ⭐⭐⭐ | ⭐         |

## 6. このプロジェクトでの推奨運用まとめ

### ブランチ戦略：GitHub Flow + タグリリース

```
main                    ← これ1本でOK
  ├─ feature/login      ← 機能開発（短命）
  ├─ feature/map        ← 機能開発（短命）
  └─ fix/crash-bug      ← バグ修正（短命）
```

### 環境への出し分け：CI/CD で自動化

```
[ main ブランチ 1本 ]
        │
        ├─ feature/xxx（短命ブランチ、数日以内にマージ）
        │
        ▼
   CI/CD パイプライン
        │
   ┌────┼────┐
   ▼    ▼    ▼
  dev  stg  prod   ← Flavor + 環境変数で出し分け
```

### GitHub Actions サンプル（`.github/workflows/deploy.yml`）

```yaml
name: Deploy

on:
  push:
    branches: [main] # main への push → stg
    tags: ["v*.*.*"] # タグ push → prod

jobs:
  deploy-stg:
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: subosito/flutter-action@v2
      - run: flutter build apk --flavor stg -t lib/main_stg.dart
      # → Firebase App Distribution などで配信

  deploy-prod:
    if: startsWith(github.ref, 'refs/tags/v')
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: subosito/flutter-action@v2
      - run: flutter build appbundle --flavor prod -t lib/main_prod.dart
      # → Google Play Console にアップロード
```

### 役割整理

|                       | 役割                          |
| --------------------- | ----------------------------- |
| Git ブランチ・タグ    | コード履歴の管理              |
| Flavor (dev/stg/prod) | ビルド設定の切り替え          |
| CI/CD                 | ブランチ→環境への自動デプロイ |

> **「ブランチで環境を分ける」のではなく、「ブランチの状態 = どの環境にデプロイするか」を CI/CD で決める**
